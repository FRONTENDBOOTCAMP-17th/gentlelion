# Gentlelion

> 젠틀몬스터 공식 홈페이지 클론 코딩 프로젝트

<br />

## 📌 프로젝트 소개

**Gentlelion**은 바닐라 JavaScript를 중심으로 젠틀몬스터 홈페이지를 클론 코딩한 팀 프로젝트입니다.

단순히 화면을 따라 만드는 것에서 그치지 않고, **유저 페이지와 관리자 페이지를 분리 배포하는 모노레포 구조**로 실제 서비스에 가까운 완성도를 목표로 했습니다.

- **기간:** 2026년 3월 26일 ~ 4월 20일
- **유저 페이지:** [배포 링크](https://gentlelion.vercel.app/)
- **어드민 페이지:** [배포 링크](https://gentlelion-adminmin.vercel.app/)

### 프로젝트 필요성

바닐라 JavaScript만으로 실제 서비스 수준의 웹을 구현해보는 것이 이 프로젝트의 핵심 이유였습니다.
리액트나 Vue 같은 프레임워크를 쓰면 쉽게 해결되는 것들을 직접 손으로 해결하면서, 브라우저가 어떻게 동작하는지, DOM을 어떻게 다뤄야 하는지, 비동기 흐름을 어떻게 제어해야 하는지를 몸으로 익히고 싶었습니다. REST API 연동과 CRUD 전체 흐름을 직접 설계하고 구현하는 경험, 그리고 팀으로 협업하면서 코드 컨벤션과 공통 컴포넌트 설계가 왜 중요한지를 실제로 체감하는 것이 목표였습니다.

### 주요 학습 포인트

**바닐라 JS 모듈화**
HTML 템플릿을 `fetch`로 불러와 DOM에 주입하는 방식으로 헤더, 푸터, 상품 카드, 쇼핑백 네비게이션 등을 재사용 가능한 모듈로 만들었습니다.

**REST API 연동 및 CRUD**
상품 조회(GET), 장바구니 추가(POST), 수량 변경(PUT), 삭제(DELETE) 전체 흐름을 직접 구현했습니다. 공통 API Wrapper(`share-api`)를 설계해 인증 토큰 주입과 에러 처리를 한 곳에서 관리했습니다.

**비동기 처리와 실행 순서 제어**
`async/await` 기반으로 API 호출 순서를 제어하고, 헤더 로드 완료 후 메뉴 애니메이션을 실행하는 등 비동기 의존성을 직접 다뤄봤습니다.

**모노레포 구조 설계**
유저 페이지, 관리자 페이지, 공유 API를 하나의 레포에서 분리 관리하는 Vite 워크스페이스 구조를 직접 설정했습니다.

**Playwright 자동화 테스트**
기능 완성 후 페이지별 E2E 테스트를 작성해 회귀 버그를 방지하는 경험을 했습니다.

### 공동 목표

- 팀원들 간의 원활한 의사소통
- 코드 컨벤션 유지를 위한 지속적인 노력
- 공통 컴포넌트와 디자인 요소 통일
- 트러블 슈팅 발생 시 팀원 모두가 해결책 제시
- 기능 명세서, 문서화 전략 수립

### 핵심 목표

- REST API 기반 데이터 처리
- CRUD 흐름 이해 및 구현
- HTML 컴포넌트 분리 — 헤더, 푸터, 상품 카드 등 반복되는 마크업을 별도 파일로 분리하고 fetch로 동적 주입하는 구조
- 실제 서비스와 유사한 UI/UX 구현


## 🤝 우리의 협업 방식

### 어떻게 소통했나

매일 디스코드로 **데일리 스크럼**을 진행했습니다. 당일 작업 내용과 막히는 부분을 공유하고, 각자의 진행 상황을 GitHub Wiki에 날짜별로 기록했습니다. 주간 단위로는 한 자리에 모여 전체 진행 상황을 점검하고 다음 주 목표를 함께 설정했습니다.

### 역할은 어떻게 나눴나

첫날(3/26) 팀원 모두가 모여 Figma로 화면을 분석하고, WBS와 기능 명세서를 작성하며 각자의 책임 범위를 명확히 했습니다. 기능 명세서에는 담당자, 우선도, 완료 여부를 함께 기록해 누가 무엇을 맡았는지 항상 한눈에 볼 수 있도록 했습니다.

### 의견 충돌이 있었을 때

1주차에는 공통 컴포넌트를 각자 만들다 보니 스타일과 구조가 맞지 않아 합칠 때 충돌이 발생했습니다. 이를 계기로 팀 전체가 모여 **공통 컴포넌트 규칙**을 명문화하고, 앞으로는 규칙을 먼저 합의한 뒤 작업하기로 약속했습니다. 또한 포인트 API 동작 방식에 대해 팀원마다 다르게 이해하던 부분은 실제 테스트를 통해 확인한 뒤 팀 전체에 공유해 같은 실수가 반복되지 않도록 했습니다.

### Git으로 협업하기

브랜치 전략과 커밋 컨벤션을 처음부터 명확히 정해두고 시작했습니다. `feature/이슈번호-설명` 형태로 브랜치를 만들고, PR을 통해 코드 리뷰를 진행했습니다. 코드 리뷰에서 나온 피드백은 이슈로 등록해 추적 가능하게 관리했습니다.

| 브랜치                  | 설명             |
| ----------------------- | ---------------- |
| `main`                  | 배포 브랜치      |
| `develop`               | 통합 브랜치      |
| `feature/이슈번호-설명` | 기능 개발 브랜치 |

| 타입       | 설명             |
| ---------- | ---------------- |
| `feat`     | 새로운 기능 추가 |
| `fix`      | 버그 수정        |
| `docs`     | 문서 수정        |
| `design`   | UI/스타일 수정   |
| `refactor` | 코드 리팩토링    |
| `chore`    | 빌드, 설정 변경  |

## 🔥 트러블슈팅

<details>
<summary><b>배포 환경 경로 문제</b></summary>

**문제:** 네비게이션 컴포넌트를 동적으로 DOM에 삽입하는 방식으로 구현했는데, 개발 환경(`localhost`)과 배포 환경(Vercel)의 루트 경로가 달라 컴포넌트를 불러오지 못했습니다.

**해결:** `window.location.origin`을 활용해 현재 환경의 기준 경로를 동적으로 계산하고, `vite.config.js`에서 올바른 경로를 지정해 해결했습니다.

</details>

<details>
<summary><b>포인트 API 동작 방식 오해</b></summary>

**문제:** API 문서에 "양수면 지급, 음수면 차감"으로 명시되어 있어 변경량만 전달했으나, 실제로는 최종값을 덮어씌우는 방식이었습니다.

**해결:** 현재 포인트 + 변경량을 계산한 최종값을 전달하도록 수정했습니다. 이후 API 연동 시에는 반드시 실제 동작을 직접 테스트하는 습관을 팀 전체가 갖게 됐습니다.

</details>

<details>
<summary><b>비동기 실행 순서 충돌</b></summary>

**문제:** `layout.js`가 async로 헤더를 주입하는데, `menuAnimation.js`가 헤더 로드 완료 전에 실행되어 `#mobileMenu` 요소를 찾지 못해 null 오류가 발생했습니다.

**해결:** `loadHeader()` 호출이 완료된 이후에 `initMenuAnimation()`을 호출하도록 실행 순서를 조정했습니다.

</details>

<details>
<summary><b>상품 카드 공통 컴포넌트 분리</b></summary>

**문제:** 각 페이지마다 상품 카드 코드가 중복 작성되어 있었고 로컬 더미 데이터만 사용했습니다.

**해결:** `productCard.js`, `moreButton.js`, `layout.js`로 분리해 어떤 페이지에서든 API 데이터를 받아 동적으로 상품 목록을 생성할 수 있도록 공통화했습니다.

</details>

<details>
<summary><b>관리자 네비게이션 동적 생성</b></summary>

**문제:** 공통 컴포넌트로 만든 사이드바 코드를 재사용하려 했으나 마크업 구조가 맞지 않아 오류가 계속 발생했습니다.

**해결:** `index.js`에 네비게이션 동적 생성 함수를 별도로 만들어 모든 관리자 페이지에서 공통으로 사용할 수 있도록 재설계했습니다.

</details>

<details>
<summary><b>메인 페이지 비디오/이미지 프로그레스 바 끊김 현상</b></summary>

**문제:** 비디오와 달리 이미지는 재생 시간 개념이 없기 때문에 프로그레스 바를 움직일 시간 데이터가 존재하지 않았습니다. `setInterval`을 사용하려 했지만 프레임이 맞지 않아 프로그레스 바가 뚝뚝 끊기며 차오르는 현상이 발생했습니다.

**해결:** 이미지는 `requestAnimationFrame`을 활용해 전용 타이머를 구현하고, 비디오는 `timeupdate` 이벤트를 사용해 실제 재생 시간과 진행률을 1:1로 매칭했습니다.

**결과:** 비디오와 이미지 모두 프로그레스 바가 끊김 없이 부드럽게 동작하게 됐습니다.

</details>
