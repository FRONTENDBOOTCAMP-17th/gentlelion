## 1. 이번에 잘한 점

### 1-1. 상품 상세 페이지의 큰 인라인 스크립트를 분리한 점

`packages/user/pages/product-detail.html`에 있던 긴 `<script type="module">` 블록을 `packages/user/components/product-detail/init.js`로 옮긴 것은 아주 좋은 방향입니다.

이전에는 HTML 파일이

- 마크업을 만들고
- 데이터를 불러오고
- 이벤트를 연결하고
- 비즈니스 로직까지 처리하고 있었어요.

지금은 적어도 “페이지 틀”과 “초기화 로직”이 분리되기 시작했습니다. 이것이 바로 **응집도를 높이는 첫걸음**입니다.

### 1-2. 상품 색상 입력 행을 공용 함수로 묶은 점

`packages/admin/pages/components/product/add.js`와 `edit.js`에 있던 비슷한 코드가 `colorRow.js`로 분리됐습니다.

이건 아주 좋은 리팩토링이에요.

- 같은 코드를 두 번 쓰지 않게 되었고
- 수정 포인트가 한 곳으로 줄었고
- `add.js`와 `edit.js`가 “색상 행을 어떻게 만드는지”까지 알 필요가 줄었습니다.

즉, **결합도는 낮아지고 응집도는 올라갔습니다.**

### 1-3. `orderSummation.js`에서 인라인 스타일 대신 Tailwind 클래스 중심으로 바꾼 점

`maxHeight`, `overflow`, `transition`을 직접 `style`로 넣는 대신 class 기반으로 바꾼 것은 방향이 좋습니다.

이 프로젝트의 원칙이 **Tailwind 우선, CSS 최소화**인 만큼, 이런 방향은 계속 유지하면 좋습니다.

### 1-4. Playwright 테스트 기반을 추가한 점

`playwright.config.js`와 `tests/` 폴더를 추가해 E2E 테스트를 시작한 것은 팀 전체에 큰 자산입니다.

초보 팀일수록 테스트가 있으면

- “내가 고친 게 어디를 깨뜨렸는지”
- “같은 버그가 또 생겼는지”

를 훨씬 빨리 확인할 수 있어요.

### 2-1. 모바일 메뉴를 HTML도 만들고 JS도 또 만들고 있어요 (중요도: 높음)

**관련 파일**

- `packages/user/index.html` 13~16줄
- `packages/user/pages/product-detail.html` 18~21줄
- `packages/user/components/mobilemenu/createMobileMenu.js` 48~92줄

현재 `index.html`, `product-detail.html` 같은 페이지에는 이미 아래처럼 `#mobileMenu`가 있습니다.

```html
<div
  id="mobileMenu"
  class="fixed opacity-0 pointer-events-none transition-all duration-500 top-0 z-49 left-0 bg-white w-screen h-full"
></div>
```

그런데 `createMobileMenu.js`에서는 또 다시

```js
const mobileMenu = document.createElement("div");
mobileMenu.id = "mobileMenu";
document.body.prepend(mobileMenu);
```

를 실행하고 있어요.

#### 왜 문제일까요?

이건 **역할이 겹친 상태**입니다.

- HTML도 “모바일 메뉴 컨테이너를 만든다”
- JS도 “모바일 메뉴 컨테이너를 만든다”

즉, **누가 책임자인지 불분명**합니다.

이렇게 되면

- 같은 `id`가 페이지에 두 번 생길 수 있고
- `getElementById("mobileMenu")`가 어느 요소를 잡을지 예측이 어려워지고
- 메뉴 애니메이션 코드가 다른 요소를 잡을 가능성이 생깁니다.

초보 단계에서 자주 생기는 문제인데, 핵심은 아주 단순해요.

> **하나의 요소는 한 곳에서만 만든다.**

#### 응집도/결합도 관점에서 보면

- 지금은 페이지 HTML과 메뉴 생성 JS가 서로의 내부 구조를 동시에 책임지고 있어서 **결합도가 높습니다.**
- 메뉴 컨테이너 생성 책임을 한쪽으로 몰아주면, 메뉴 관련 변경이 메뉴 코드 안에서 끝나므로 **응집도가 높아집니다.**

#### 추천 방향

둘 중 하나만 선택하세요.

1. HTML은 빈 컨테이너만 두고, JS는 **그 안을 채우기만** 한다.
2. HTML에서 `#mobileMenu`를 없애고, JS가 **처음부터 끝까지 전부 만든다.**

이번 코드베이스에서는 이미 여러 페이지에 빈 컨테이너가 있으니, **1번 방식이 더 안전하고 쉬워 보입니다.**

> 자세한 해결 방법은 [2026041601_리뷰_해결.md](./2026041601_리뷰_해결.md)의 **2-1** 참고

### 2-2. `wishlistApi.js`의 에러 처리에서 정의되지 않은 `APIError`를 사용하고 있어요 (중요도: 높음)

**관련 파일**

- `packages/user/components/API/wishlist/wishlistApi.js` 9~11줄

현재 코드는 이렇게 되어 있습니다.

```js
return get("/wishlist").catch((err) => {
  if (err instanceof APIError && err.status === 401) return [];
  throw err;
});
```

그런데 이 파일 안에는 `APIError`를 import한 코드가 없습니다. 즉, 실제 에러가 발생하면 원래 에러를 처리하기 전에 **`APIError is not defined`** 같은 새로운 에러가 생길 수 있어요.

#### 왜 문제일까요?

이건 단순 문법 문제가 아니라, **에러의 원인을 숨기는 코드**가 됩니다.

예를 들어 실제 원인이

- 토큰 만료인지
- 서버 500 에러인지
- 네트워크 문제인지

알아야 하는데, 화면에는 엉뚱한 `APIError` 관련 에러만 보일 수 있습니다.

초보 개발자에게 가장 위험한 코드 중 하나가 이런 코드예요.

> **원래 문제보다, 에러 처리 코드가 더 큰 문제를 만드는 경우**

#### 응집도/결합도 관점에서 보면

`wishlistApi.js`는 지금 “위시리스트 요청”만 알면 되는데, 존재하지도 않는 `APIError` 타입을 전제로 작성되어 있어요. 즉, 이 파일이 외부 구현 세부사항에 잘못 결합돼 있습니다.

#### 추천 방향

- `err.status === 401`처럼 실제 지금 쓰는 에러 객체 구조만 이용하세요.
- 또는 `share-api`에서 공통 에러 타입을 정말 만들 거라면, 그 타입을 **명확하게 export/import** 하세요.

지금 단계에서는 **단순하게 `status`만 검사하는 방식**이 가장 이해하기 쉽습니다.

> 자세한 해결 방법은 해결 가이드 **2-2** 참고

### 2-3. 비로그인 상태에서 위시리스트 버튼을 눌렀을 때, 리다이렉트 후에도 함수가 계속 실행돼요 (중요도: 높음)

**관련 파일**

- `packages/user/components/productCard/wishbuttonEvent.js` 10~19줄, 25~30줄

현재 코드는 비로그인 상태일 때

```js
if (!data) {
  alert("로그인이 필요한 기능입니다.");
  window.location.href = "/components/login/login.html";
}
```

까지만 하고 끝내지 않습니다. `return`이 없어서 그 아래 코드가 계속 실행돼요.

그 결과로 뒤에서

- `postWishlist(productId, color)`를 호출하거나
- `data.data.items.find(...)`를 실행하려고 하게 됩니다.

#### 왜 문제일까요?

로그인 페이지로 보내겠다고 해놓고, 실제로는 현재 페이지에서 API 요청이나 데이터 접근을 더 시도하는 셈입니다.

이건 코드가 스스로 한 약속을 지키지 못하는 상태예요.

- 사용자 입장에서는 경고를 봤는데 추가 에러가 날 수 있고
- 개발자 입장에서는 “왜 redirect 했는데도 에러가 나지?” 같은 혼란이 생깁니다.

#### 응집도/결합도 관점에서 보면

이 함수는 “로그인 여부 판단”, “위시리스트 등록/삭제”, “버튼 모양 변경”을 모두 하고 있습니다. 이미 역할이 많은 편인데, 비로그인 흐름까지 끝내지 않고 중간에 흘려보내서 함수의 책임 경계가 더 흐려졌어요.

#### 추천 방향

비로그인 분기에서는 **즉시 종료(return)** 해야 합니다.

가능하면 아래처럼 분기를 작게 잘라서 읽기 쉽게 만드세요.

- `if (!data) return handleRequireLogin();`
- 그 아래는 “로그인된 상태에서만 실행되는 코드”만 남기기

이렇게 하면 함수 읽기가 훨씬 쉬워집니다.

> 자세한 해결 방법은 해결 가이드 **2-3** 참고

### 2-4. 테스트 결과 파일을 저장소에 함께 올리면 팀 작업이 지저분해져요 (중요도: 중간)

**관련 파일**

- `playwright-report/index.html`
- `test-results/.last-run.json`
- `.gitignore`

이번 변경에서 Playwright 설정을 추가한 것은 아주 좋았습니다. 그런데 동시에 **로컬 실행 결과물**도 함께 추적되고 있어요.

`playwright-report/`는 이번에 `.gitignore`에 추가됐지만, 이미 추적된 파일은 자동으로 빠지지 않습니다. 그래서 지금처럼 `playwright-report/index.html`과 `test-results/.last-run.json`이 계속 저장소에 남게 됩니다.

#### 왜 문제일까요?

이 파일들은 “소스 코드”가 아니라 “실행 결과”입니다.

즉, 팀원이 테스트를 한 번만 다시 돌려도

- 커밋 diff가 시끄러워지고
- 리뷰어가 진짜 코드 변경을 찾기 어려워지고
- PR 품질이 떨어집니다.

초보 팀일수록 이 차이를 꼭 알아야 해요.

> **코드와 결과물은 다릅니다. 결과물은 보통 버전 관리에서 제외합니다.**

#### 추천 방향

- `playwright-report/`
- `test-results/`

둘 다 `.gitignore`에 넣고, 이미 추적 중인 파일은 인덱스에서 제거하세요.

## 3. 이번 리뷰의 핵심 정리

이번 변경은 전체적으로 **“중복 코드 줄이기”와 “페이지 스크립트 분리”라는 좋은 방향**으로 가고 있습니다. 특히 `product-detail` 초기화 분리, `colorRow` 공통화, Playwright 도입은 확실히 칭찬할 만합니다.

다만 아직 몇 군데는 다음 문제가 남아 있어요.

- **한 요소를 HTML과 JS가 동시에 관리함**
- **에러를 처리하려다 오히려 새로운 에러를 만듦**
- **리다이렉트 분기에서 함수 종료가 빠짐**
- **실행 결과물을 소스처럼 커밋함**

이 네 가지는 전부 “코드가 어디까지 책임지는지”를 분명히 하면 해결됩니다.

초보 개발자에게 가장 중요한 습관은 이것입니다.

> **한 파일은 한 가지 이유로만 바뀌게 만들기**

이 기준으로 계속 리팩토링하면, 응집도는 올라가고 결합도는 자연스럽게 내려갑니다.
