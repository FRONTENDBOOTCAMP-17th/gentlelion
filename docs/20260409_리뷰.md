## 1. 치명적 버그 (즉시 수정 필요)

### 1-1. 토큰 키 불일치 - 상품 관리 기능 동작 불가

**문제**: 로그인 시 `"token"`이라는 이름으로 토큰을 저장하는데, 상품 관련 API에서는 `"admin_token"`이라는 이름으로 토큰을 읽고 있습니다. 이름이 다르기 때문에 **상품 페이지는 항상 인증 실패**합니다.

| 파일 | 코드 | 문제 |
|------|------|------|
| `admin/pages/API/login/loginApi.js:19` | `localStorage.setItem("token", data.token)` | `"token"`으로 저장 |
| `admin/pages/API/product/productListApi.js:3` | `localStorage.getItem("admin_token")` | `"admin_token"`으로 읽음 |
| `admin/pages/API/product/productAdd.js:4` | `localStorage.getItem("admin_token")` | `"admin_token"`으로 읽음 |
| `admin/pages/API/product/productEditApi.js:2` | `localStorage.getItem("admin_token")` | `"admin_token"`으로 읽음 |
| `admin/pages/API/product/productDelete.js:2` | `localStorage.getItem("admin_token")` | `"admin_token"`으로 읽음 |
| `admin/pages/API/product/imageApi.js:4` | `localStorage.getItem("admin_token")` | `"admin_token"`으로 읽음 |

**왜 심각한가**: 로그인은 성공하지만 상품 목록 조회, 등록, 수정, 삭제 모든 기능이 인증 실패로 동작하지 않습니다.

### 1-2. main.js에서 정의되지 않은 변수 사용

**파일**: `src/main.js:8`

```javascript
if (btn && line1 && line2 && header) {  // header가 정의되지 않음!
```

`header` 변수가 이 파일에서 선언된 적이 없습니다. 따라서 이 조건문은 항상 에러를 발생시키거나, 전역 변수가 없으면 메뉴 버튼이 동작하지 않습니다.

### 1-3. dashboard.html 잘못된 HTML 구조

**파일**: `admin/pages/dashboard.html:82`

```html
<tbody id="orderTableBody">
  <td>안녕</td>   <!-- <tr>로 감싸지 않은 <td> -->
</tbody>
```

`<td>`는 반드시 `<tr>` 안에 있어야 합니다. 브라우저가 임의로 해석하므로 레이아웃이 깨질 수 있습니다.

## 2. 코드 중복 문제

### 2-1. createColor() 함수 완전 복사-붙여넣기

**결합도 높음, 응집도 낮음** - 같은 기능이 두 곳에 분산

| 파일 | 라인 |
|------|------|
| `admin/pages/components/product/add.js` | 39~88행 |
| `admin/pages/components/product/edit.js` | 9~58행 |

이 두 함수는 **거의 100% 동일**합니다. 색상 입력 필드를 만들고, 삭제 버튼을 붙이고, 이벤트를 연결하는 로직이 완전히 같습니다.

**왜 나쁜가**:
- 버그를 고칠 때 한쪽만 고치고 다른 쪽을 깜빡할 수 있습니다
- 디자인을 바꿀 때 두 곳을 모두 수정해야 합니다
- 코드가 길어져서 읽기 어렵습니다

### 2-2. 스크롤 이벤트 로직 3곳에 중복

같은 스크롤 처리 로직이 **3개 파일**에 분산되어 있습니다:

| 파일 | 내용 |
|------|------|
| `src/components/header/header.js:12~18` | `scrollY > 785.33` 체크 |
| `src/components/header/headerEvent.js:4~9` | 동일한 코드 |
| `src/main.js:8~17` | 메뉴 버튼 이벤트 (headerEvent.js와 중복) |

**문제**: 스크롤 이벤트가 **2번 등록**될 수 있어 성능이 낭비되고, 수정할 때 어디를 고쳐야 할지 혼란스럽습니다.

### 2-3. 이미지 업로드 로직 중복

| 파일 | 라인 |
|------|------|
| `admin/pages/components/product/add.js` | 8~36행 (`imageUpload()`) |
| `admin/pages/components/product/edit.js` | 108~131행 (`EditPage()` 내부) |

이미지를 선택하고, 업로드하고, 미리보기를 표시하는 로직이 거의 동일합니다.

### 2-4. Navigation() 함수 중복

"뒤로가기" 버튼 이벤트 연결 로직이 완전히 동일합니다.

## 3. 응집도 & 결합도 문제

### 3-1. 응집도가 낮은 파일 - add.js

**파일**: `admin/pages/components/product/add.js` (178행)

이 파일 하나가 **4가지 서로 다른 역할**을 합니다:

1. 이미지 업로드 처리 (8~36행)
2. 색상 입력 관리 (39~113행)
3. 페이지 네비게이션 (115~124행)
4. 폼 제출 처리 (137~175행)

**응집도가 높다**는 것은 하나의 파일이 **하나의 역할에 집중**한다는 뜻입니다. 이 파일은 너무 많은 일을 하고 있어서 응집도가 낮습니다.

### 3-2. 결합도가 높은 API 파일들

모든 API 파일이 토큰 가져오기, URL 구성, 에러 처리를 각자 직접 하고 있습니다:

```
admin/pages/API/product/productListApi.js  → localStorage.getItem(...), fetch(URL, ...), try/catch
admin/pages/API/product/productAdd.js      → localStorage.getItem(...), fetch(URL, ...), try/catch
admin/pages/API/product/productEditApi.js  → localStorage.getItem(...), fetch(URL, ...), try/catch
admin/pages/API/product/productDelete.js   → localStorage.getItem(...), fetch(URL, ...), try/catch
admin/pages/API/product/imageApi.js        → localStorage.getItem(...), fetch(URL, ...), try/catch
admin/pages/API/login/loginApi.js          → fetch(URL, ...), try/catch
admin/pages/API/login/logoutApi.js         → localStorage.getItem(...), fetch(URL, ...), try/catch
```

**왜 나쁜가**: API URL이 바뀌면 모든 파일을 수정해야 합니다. 토큰 키 이름이 바뀌면(지금도 불일치 상태!) 모든 파일을 찾아야 합니다.

### 3-3. window 전역 스코프 오염

여러 파일에서 함수를 `window` 객체에 직접 붙이고 있습니다:

| 파일 | 코드 |
|------|------|
| `admin/pages/components/login/handleLogin.js:26` | `window.handleLogin = handleLogin` |
| `admin/pages/components/dashboard/index.js:6~7` | `window.toggleMenu`, `window.logout` |
| `admin/pages/components/userDetails/index.js:18` | `window.adjustPoint = ...` |

**왜 나쁜가**: 
- 다른 파일의 코드가 같은 이름을 쓰면 덮어써질 수 있습니다
- 어디서 이 함수가 호출되는지 추적하기 어렵습니다
- HTML에서 `onclick="함수이름()"` 방식은 오래된 패턴입니다

### 3-4. 하드코딩된 파일 경로

컴포넌트들이 HTML 파일을 불러올 때 절대 경로를 하드코딩하고 있습니다:

| 파일 | 경로 |
|------|------|
| `src/components/header/header.js:2` | `'/src/components/header/header.html'` |
| `src/components/footer/footer.js:2` | `'/src/components/footer/footer.html'` |
| `admin/pages/components/aside/dashboardNavigation.js:5` | `'/admin/pages/components/aside/sidebar.html'` |

파일 위치를 변경하면 코드가 깨집니다.

## 4. CSS 사용 문제 (TailwindCSS 미준수)

### 4-1. JavaScript에서 인라인 스타일 과다 사용

**가장 심각한 파일**: `src/components/item/item.js`

이 파일은 260행에 걸쳐 거의 모든 스타일을 JavaScript의 `style.cssText`로 작성하고 있습니다:

```javascript
// item.js:6~8
imageWrap.style.cssText =
  "position:relative;width:100%;aspect-ratio:520 / 718;overflow:hidden;" +
  "background:#F3F4F6;cursor:grab;user-select:none;-webkit-user-select:none;";

// item.js:225~226
infoWrap.style.cssText =
  "display:flex;align-items:flex-start;justify-content:space-between;padding:8px 0 24px;";
```

**TailwindCSS로 작성해야 하는 이유**:
- CSS를 한 곳에서 관리할 수 있습니다
- 클래스 이름만 보고 스타일을 바로 파악할 수 있습니다
- 반응형 디자인을 `md:`, `lg:` 접두사로 쉽게 적용할 수 있습니다

### 4-2. PR #122의 imageCarousel.js도 같은 문제

새로 분리한 `src/components/carousel/imageCarousel.js`도 모든 스타일을 `style.cssText`로 작성하고 있습니다 (189행 전체). 분리한 것은 좋은 시도이지만, 스타일링 방식은 개선이 필요합니다.

### 4-3. 커스텀 CSS가 필요 이상으로 많음

**파일**: `src/components/header/header.css` (74행)

```css
.nav-bar {
  @apply text-[14px] px-2 py-1 bg-transparent border-none cursor-pointer;
  color: var(--header-icon-color);
}
```

`@apply`로 TailwindCSS 클래스를 CSS 파일에 쓰는 것은 TailwindCSS의 장점을 상쇄합니다. HTML에서 직접 클래스를 사용하거나, CSS 변수가 꼭 필요한 경우에만 최소한의 커스텀 CSS를 작성해야 합니다.

### 4-4. 색상값 하드코딩

TailwindCSS 설정 파일(tailwind.config.js)에 프로젝트 색상을 정의하지 않고, 곳곳에서 임의 값을 사용하고 있습니다:

```
bg-[#0a0a0a]      → products.html, dashboardNavigation.js
border-[#e4e7ec]   → products.html
text-[#d92d20]     → add.js, edit.js
background:#F3F4F6 → item.js
background:#e8e9eb → item.js
```

## 6. 매직 넘버 & 하드코딩

### 6-1. 스크롤 임계값 785.33

```javascript
// header.js:13, headerEvent.js:5
if (window.scrollY > 785.33) {
```

`785.33`이라는 숫자가 무엇을 의미하는지 코드만 보고는 알 수 없습니다. 메인 비디오 높이와 관련된 것으로 보이지만, 화면 크기가 바뀌면 이 값도 바뀌어야 하므로 하드코딩하면 안 됩니다.

### 6-2. 페이지 크기 20이 여러 곳에 분산

```javascript
// admin/pages/components/product/index.js:10
const limit = 20;

// admin/pages/components/user/index.js:9
const data = await userApi(page, 20);

// admin/pages/components/common/calcListNum.js
const pageSize = 20;
```

같은 값이 3곳 이상에서 각각 정의되어 있습니다. 하나를 바꾸면 나머지도 모두 찾아서 바꿔야 합니다.

### 6-3. API URL 하드코딩

모든 API 파일에 `https://api.fullstackfamily.com/...` URL이 직접 들어가 있습니다. 개발 환경과 운영 환경의 URL이 다를 수 있으므로, 설정 파일에서 관리해야 합니다.

## 7. PR #122 리뷰 (이미지 캐러셀 분리)

### 잘한 점
- `item.js`에서 캐러셀 로직을 `carousel/imageCarousel.js`로 분리한 것은 **좋은 리팩토링**입니다
- 함수 하나(`createImageCarousel`)로 깔끔하게 감싼 것이 좋습니다
- `item.js`가 260행에서 크게 줄어드는 효과가 있습니다

### 개선 필요한 점

1. **인라인 스타일이 그대로 옮겨졌을 뿐**
   - `imageCarousel.js` 전체가 `style.cssText` 문자열입니다
   - TailwindCSS 클래스(`className`)를 사용하도록 바꿔야 합니다

2. **`colorChips.js`도 인라인 스타일 사용**
   ```javascript
   swatch.style.cssText = `width:24px;height:24px;background:${color.background};`;
   ```
   `className = "w-6 h-6"` + 동적 배경색만 style로 처리하는 것이 좋습니다

3. **`colors.js` 데이터 파일은 적절**
   - 색상 데이터를 별도 파일로 분리한 것은 좋습니다

## 8. 잘한 점

모든 것이 개선 필요한 것은 아닙니다. 아래는 이번 코드에서 잘 된 부분입니다:

### 컴포넌트 폴더 구조화
```
admin/pages/
├── API/           → API 호출 분리
├── components/
│   ├── common/    → 공통 유틸리티
│   ├── product/   → 상품 관련
│   ├── user/      → 사용자 관련
│   ├── order/     → 주문 관련
│   └── dashboard/ → 대시보드 관련
```
기능별로 폴더를 나눈 것은 훌륭합니다.

### 공통 컴포넌트 추출
- `common/renderRows.js` - 테이블 행 렌더링을 설정(config) 방식으로 일반화
- `common/search.js` - 검색 기능을 재사용 가능하게 만듦
- `common/calcListNum.js` - 목록 번호 계산 공통화

### 이벤트 위임 패턴 사용
```javascript
// admin/pages/components/product/index.js:83
tbody.addEventListener("click", async (e) => {
  const deleteBtn = e.target.closest("#deleteBtn");
  if (!deleteBtn) return;
  // ...
});
```
테이블의 각 버튼에 이벤트를 붙이는 대신, 부모 요소에서 이벤트를 위임하는 패턴을 사용한 것이 좋습니다.

### ES 모듈 사용
`import`/`export` 구문을 활용해 모듈 단위로 코드를 관리하고 있습니다.

### 사이드바 네비게이션 자동 활성화
```javascript
// dashboardNavigation.js:9~16
const currentPath = window.location.pathname;
if (currentPath.includes("dashboard")) activeNav = "dashboard";
```
현재 페이지에 맞는 네비게이션 항목을 자동으로 하이라이트하는 로직이 깔끔합니다.
