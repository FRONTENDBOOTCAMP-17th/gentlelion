## 심각 (반드시 수정)

### 1. register.html — 체크박스 id가 전부 같습니다

**파일**: `src/components/login/register.html`

```html
<!-- 5개의 체크박스가 전부 id="custom-checkbox" -->
<input type="checkbox" id="custom-checkbox" ... />
<input type="checkbox" id="custom-checkbox" ... />
<input type="checkbox" id="custom-checkbox" ... />
```

**왜 문제인가요?**

HTML에서 `id`는 **페이지 안에서 딱 하나만** 있어야 합니다. 같은 id가 여러 개 있으면:

- `document.getElementById("custom-checkbox")`가 첫 번째 요소만 찾아줍니다
- `<label for="custom-checkbox">`를 클릭해도 첫 번째 체크박스만 반응합니다
- 접근성(스크린 리더)도 제대로 동작하지 않습니다

또한 `<label for="checked-checkbox">`처럼 **존재하지 않는 id**를 참조하는 label도 있어서, 라벨을 클릭해도 체크박스가 반응하지 않습니다.

### 2. goToMyPage.js — getProfileApi, getToken을 쓰지 않고 직접 fetch

**파일**: `src/components/header/goToMyPage.js`

```javascript
// 이미 getProfileApi가 있는데 직접 fetch를 호출합니다
const response = await fetch("https://api.fullstackfamily.com/api/gentlelion/v1/user/profile", {
    method: "GET",
    headers: { "Authorization": `Bearer ${token}` }
});

// 이미 getToken()이 있는데 직접 localStorage를 접근합니다
const token = localStorage.getItem("token");
```

**왜 문제인가요?**

같은 일을 하는 코드가 2곳에 있으면 **한쪽만 고치고 다른 쪽을 깜빡**하게 됩니다. 예를 들어:

- API 주소가 바뀌면 `getProfileApi.js`와 `goToMyPage.js` 두 곳 다 수정해야 합니다
- 토큰 저장 방식이 바뀌면 `getToken.js`와 `goToMyPage.js` 두 곳 다 수정해야 합니다

이것을 **높은 결합도(coupling)** 라고 합니다. 하나를 바꿨을 때 다른 곳도 같이 바꿔야 하는 상황이죠.

## 개선 권장

### 4. HTML 모바일 메뉴가 모든 페이지에 복사-붙여넣기 되어 있습니다

**파일**: `order.html`, `login.html`, `register.html` 모두 동일한 mobileMenu 코드(약 40줄)가 복사되어 있습니다.

```html
<!-- 이 코드가 order.html, login.html, register.html에 전부 똑같이 있습니다 -->
<div id="mobileMenu" class="fixed opacity-0 pointer-events-none ...">
  <aside class="relative top-40 ml-5">
    <nav>
      <ul class="font-medium flex flex-col gap-2 mb-3">
        <li class="menu-animate text-[22px]"><a href="/">선글라스</a></li>
        <!-- ... 약 35줄 더 ... -->
      </ul>
    </nav>
  </aside>
</div>
```

**왜 문제인가요?**

- 메뉴 항목을 하나 추가하려면 **3개 파일을 전부 수정**해야 합니다
- 한 파일만 수정하고 나머지를 깜빡하면 페이지마다 메뉴가 달라집니다
- 이것을 **낮은 응집도**라고 합니다 — 메뉴라는 하나의 기능이 여러 파일에 흩어져 있는 상태입니다

프로젝트에 이미 `layout.js`가 `loadHeader()`를 호출해서 헤더를 로드하고 있습니다. 모바일 메뉴도 이 방식으로 한 곳에서 관리해야 합니다.

### 6. order.html — 모바일/데스크탑 섹션이 통째로 두 번 작성되어 있습니다

**파일**: `src/components/order/order.html`

이메일 주소, 배송, 결제 섹션이 데스크탑용(`hidden md:flex`)과 모바일용(`md:hidden`) 두 벌로 작성되어 있습니다.

```html

<div class="hidden md:flex ...">
  <section>이메일 주소</section>
  <section>배송</section>
```

**왜 문제인가요?**

- TailwindCSS의 반응형 클래스를 잘 활용하면 **하나의 마크업**으로 두 레이아웃 모두 처리할 수 있습니다

### 7. orderCard.js와 cartCard.js의 템플릿 로딩 패턴이 중복됩니다

두 파일 모두 이 흐름을 똑같이 반복합니다:

```javascript
// orderCard.js와 cartCard.js 모두 이 패턴
const response = await fetch('/src/components/.../card.html');
const html = await response.text();

for (const product of data.data.items) {
    const parser = new DOMParser();
    const doc = parser.parseFromString(html, 'text/html');
    const card = doc.body.firstElementChild;
    // ... 데이터 바인딩 ...
    productContainer.appendChild(card);
}
```

**왜 문제인가요?**

- HTML을 fetch → DOMParser로 파싱 → 데이터 채워넣기 패턴을 **매번 새로 작성**해야 합니다
- 반복문 안에서 매번 `new DOMParser()`를 만드는 것도 비효율적입니다
- 템플릿 로딩이라는 **하나의 역할**을 하는 코드가 여러 파일에 흩어져 있습니다

### 8. checkPriceChange.js — API 호출이 API 폴더 구조를 무시합니다

**파일**: `src/components/API/cart/checkPriceChange.js`

```javascript
// API 폴더 안에 있으면서 직접 fetch를 호출합니다
await fetch(`https://api.fullstackfamily.com/api/gentlelion/v1/cart/${product.cartItemId}/acknowledge-price`, {
    method: 'PATCH',
    headers: { 'Authorization': `Bearer ${token}` }
});
```

**왜 문제인가요?**

이 파일은 두 가지 역할을 동시에 합니다:
1. 사용자에게 가격 변동을 알리는 **UI 로직** (`confirm()` 다이얼로그)
2. 서버에 가격 인정을 알리는 **API 호출** (`fetch`)

API 폴더에 있으면서 `confirm()`과 `window.location.href`를 사용하는 것은 역할이 섞여 있는 상태입니다. API 함수는 **데이터 통신만**, UI 로직은 **컴포넌트에서** 처리해야 합니다.

## 잘한 점

리뷰에서 문제점만 이야기하면 안 되니까, 잘한 부분도 적어둡니다:

1. **API를 별도 폴더로 분리한 것** — `src/components/API/` 구조로 API 호출을 모아둔 것은 좋은 설계입니다
2. **HTML 템플릿 분리** — `orderCard.html`, `cartCard.html`처럼 카드 템플릿을 별도 파일로 분리한 것은 재사용성을 높여줍니다
3. **IntersectionObserver 활용** — `orderNavigation.js`에서 스크롤 위치에 따라 하단 바를 보여주는 처리가 깔끔합니다
4. **cartAccordion 재사용** — order 페이지에서 cart의 아코디언 컴포넌트를 import해서 재사용한 것은 좋은 판단입니다
5. **layout.js로 공통 초기화 통합** — 헤더, 푸터, 이벤트를 한 곳에서 초기화하는 패턴이 깔끔합니다
