## 2. 잘한 점

### 2-1. `share-api`로 API 호출을 한 곳에 모은 것 — 응집도의 완벽한 예시

`packages/share-api/index.js`에서 `get`, `post`, `put`, `patch`, `del`, `upload`를 한 곳에서 정의하고, 각 패키지가 `@gentlelion/share-api`로 import해서 쓰는 구조예요.

예를 들어 `packages/user/components/API/cart/postCartApi.js`:
```js
import { post } from "@gentlelion/share-api";

export const postCartApi = (productId, quantity, color) =>
  post("/cart", { productId, quantity, color });
```

단 3줄이에요! 토큰 처리, 에러 처리, JSON 변환이 모두 `share-api` 안에서 일어나기 때문에 **API 함수 하나가 할 일이 "엔드포인트와 바디만 지정하는 것"으로 줄어들었어요.** 이게 응집도가 높다는 거예요: "하나의 파일이 한 가지 일만 한다."

### 2-2. 컴포넌트별로 폴더가 잘 나뉘어 있어요

`components/accordion/`, `components/carousel/`, `components/slide/` 처럼 **기능 단위로 폴더를 나눈 것**은 초보 단계에서 잡기 어려운 감각이에요. 각 폴더 안에 필요한 HTML, JS가 함께 있어서 "아, 여기를 고치면 되겠다"가 바로 보여요.

특히 `slide/mainSlide.js`, `slide/arrivals.js`, `slide/bestSeller.js` 처럼 **"슬라이드"라는 큰 주제 아래 여러 변형을 같은 폴더에 둔 것**은 훌륭합니다.

### 2-3. 시맨틱 태그를 잘 쓰고 있어요

`packages/user/index.html`, `collection.html` 등에서 `<header>`, `<main>`, `<section>`, `<aside>`, `<nav>`, `<figure>`, `<footer>` 같은 의미 있는 태그를 사용하고 있어요. 스크린 리더나 검색 엔진이 페이지 구조를 이해하기 쉬워집니다.

### 2-4. Vite 빌드 설정에 모든 HTML을 등록

`packages/user/vite.config.js`의 `rollupOptions.input`에 `collection`, `glasses`, `sunglasses`, `order`, `product-detail` 모두 등록되어 있어요. 이전 리뷰에서 `order.html`이 빌드에 빠져있었던 문제가 완전히 해결됐어요. 👍

### 2-5. DOM 요소 null 체크를 시작한 흔적

`components/slide/arrivals.js` 6번 줄:
```js
if (!wrapper || !thumbsWrapper) return;
```

`components/header/header.js` 11번 줄:
```js
if (!utilityBar) return;
```

`components/mobilemenu/createMobileMenu.js`에서도 null 체크가 보여요. **"요소가 없으면 그냥 조용히 돌아가기"** 패턴을 의식적으로 쓰고 있는 거예요. 이 습관을 모든 DOM 조작 파일로 확장하면 아주 좋아질 거예요 (아래 3-3 참고).

### 2-6. `handlerRegister.js`의 객체 인자 변환

이전 리뷰에서 가장 크게 지적했던 부분을 완벽히 고쳤어요:

```js
const userData = {
  email: document.getElementById("emailInput").value.trim(),
  password: document.getElementById("passwordInput").value.trim(),
  firstName: document.getElementById("firstName").value.trim(),
  // ...
};
await postRegisterApi(userData);
```

`postRegisterApi(email, pw, fn, ln, phone, addr, addrDetail)` 같은 "순서 외우기 지옥"에서 벗어난 거예요. 🎉

중요한 것부터 번호를 매겼어요. 번호가 작을수록 먼저 고치면 좋습니다.

### 3-1. `BASE_URL`이 두 곳에 하드코딩돼 있어요 (중요도: 높음)

**파일 1**: `packages/share-api/index.js` 1번 줄
```js
const BASE_URL = "https://api.fullstackfamily.com/api/gentlelion/v1";
```

**파일 2**: `packages/user/components/API/product/productDetailApi.js` 1번 줄
```js
const BASE_URL = "https://api.fullstackfamily.com/api/gentlelion/v1";

export async function fetchProduct(productId) {
  const res = await fetch(BASE_URL + "/products/" + productId);
  // ...
}
```

**왜 개선하면 좋을까?**

API 서버 주소가 바뀌면 **두 파일을 동시에 고쳐야** 해요. 한 곳만 고치면 어떤 페이지는 되고 어떤 페이지는 안 되는 이상한 버그가 생깁니다.

우리는 이미 `share-api`라는 멋진 공용 모듈을 만들었어요. 그런데 `productDetailApi.js`만 이걸 안 쓰고 있는 거예요. 마치 **공용 우산을 만들어놓고 혼자 비 맞고 가는 것**과 같아요.

**추천:** `productDetailApi.js`도 `share-api`의 `get()` 함수를 쓰도록 바꾸면 됩니다. 그럼 `BASE_URL`이 한 곳에만 존재해요.

```js
// 수정 후 예시
import { get } from "@gentlelion/share-api";

export const fetchProduct = (productId) => get(`/products/${productId}`);

export async function fetchSimilarProducts(category, productId) {
  const json = await get(`/products?category=${category}&limit=6`);
  return (json.data || [])
    .filter((p) => String(p.id) !== String(productId))
    .slice(0, 5);
}
```

**파일 1**: `packages/user/index.html` 13~56줄
**파일 2**: `packages/user/pages/product-detail.html` 13~54줄

```html
<div
  id="mobileMenu"

  <aside class="relative top-40 ml-5">
```

### 3-3. DOM 요소가 없을 때 에러가 나는 곳들 (중요도: 높음)

`document.getElementById(...)`나 `querySelector(...)`는 **요소가 없으면 `null`을 반환**해요. 그걸 바로 `.addEventListener(...)` 하면 **"Cannot read properties of null"** 에러로 페이지가 멈춰요.

**문제 지점들:**

**파일 1**: `packages/user/components/header/headerEvent.js` 3~7줄
```js
const btn = document.getElementById("menu-btn");
const line1 = document.getElementById("line1");
const line2 = document.getElementById("line2");

btn.addEventListener("click", () => {   // ← btn이 null이면 폭발!
  line1.classList.toggle(...);
});
```

**파일 2**: `packages/admin/pages/components/orderDetails/cancleButton.js` 2~4줄
```js
export function cancleButton() {
  const cancle = document.getElementById("cancle");
  cancle.addEventListener("click", () => {   // ← 같은 문제
    window.close();
  });
}
```

**파일 3**: `packages/user/components/navigation/createNavigation.js` 2줄 근처
```js
const aside = document.querySelector(".navigation");
// ... 이후 aside.appendChild(nav) 에서 에러 가능
```

**파일 4**: `packages/admin/pages/components/product/add.js` 9~13줄
```js
const uploadBtn = document.getElementById("imageUploadBtn");
const fileInput = document.getElementById("imageInput");
// ... null 체크 없이 바로 사용
```

**왜 개선하면 좋을까?**

**헤더가 없는 페이지에서 `headerEvent()`가 실행되면** (예: 로그인 페이지) 이 한 줄의 에러 때문에 **그 페이지의 나머지 JS가 전부 멈춰요.** 하나가 무너지면 다 같이 무너지는 거죠. 이게 결합도가 높다는 얘기예요.

**추천 패턴:**
```js
export function headerEvent() {
  const btn = document.getElementById("menu-btn");
  const line1 = document.getElementById("line1");
  const line2 = document.getElementById("line2");
  if (!btn || !line1 || !line2) return;   // ← 가드 한 줄 추가

  btn.addEventListener("click", () => { ... });
}
```

> 상세한 해결 방법은 `2026041509_리뷰_해결.md` **3-3** 참고

### 3-4. `cancleButton.js` 파일명과 ID 오타 — `cancle` → `cancel` (중요도: 높음)

**파일**: `packages/admin/pages/components/orderDetails/cancleButton.js` (파일명 자체가 오타)

```js
export function cancleButton() {
  const cancle = document.getElementById("cancle");
  cancle.addEventListener("click", () => {
    window.close();
  });
}
```

**왜 개선하면 좋을까?**

"cancel"이 맞고 "cancle"은 틀린 철자예요. 지금은 HTML의 `id="cancle"`과 JS의 `getElementById("cancle")`가 둘 다 틀려서 **우연히 맞아떨어져** 동작하지만, 나중에 다른 사람이 "어? 왜 cancel인데 안 되지?" 하며 HTML만 고치면 JS가 깨져요.

초보 단계에서 가장 중요한 습관 중 하나가 **"첫 번째부터 제대로 된 이름을 쓰는 것"** 입니다. 영어 단어 확신이 없으면 브라우저 번역기를 켜두는 것도 방법이에요.

**추천:** 파일명, 함수명, HTML `id` 모두 `cancel`로 바꾸세요.

```js
// cancelButton.js
export function cancelButton() {
  const cancelBtn = document.getElementById("cancelBtn");
  if (!cancelBtn) return;
  cancelBtn.addEventListener("click", () => window.close());
}
```

> 상세한 해결 방법은 `2026041509_리뷰_해결.md` **3-4** 참고

### 3-5. `product-detail.html`에 146줄짜리 `<script>` 블록이 박혀있어요 (중요도: 높음)

**파일**: `packages/user/pages/product-detail.html` 79~225줄

이전 리뷰에서 260줄짜리 script 블록을 지적했는데, 지금은 146줄로 줄었어요. 🎉 그런데 **여전히 HTML 안에 JS가 박혀 있어요.**

```html
<script type="module">
  // 상품 정보 렌더링, 슬라이더, 장바구니 추가 등
  // ... 146줄 ...
</script>
```

**왜 개선하면 좋을까?**

1. HTML 파일은 **구조**를 담고, JS 파일은 **동작**을 담는 게 원칙이에요. 한 파일에 섞이면 "여기 뭘 바꿨지?" 싶을 때 HTML 파일을 열어서 스크롤 내리다가 JS 코드 안으로 빠져버려요.
2. 이 146줄은 **재사용이 불가능**해요. 이 페이지에서만 쓸 수 있어요.
3. Git에서 변경 내역을 볼 때도, HTML 변경인지 JS 변경인지 섞여서 보여요.

**좋은 소식:** 이미 프로젝트에는 `packages/user/components/product-detail/Productdetailrenderer.js`, `addCartButton.js`가 있어요. 즉, **분리를 시작만 하고 마무리하지 못한 상태**예요. 반만 남은 작업을 끝내면 됩니다.

**추천:** HTML 안의 `<script>` 내용을 `components/product-detail/init.js`(또는 비슷한 이름) 파일로 옮기고, HTML에서는 `<script type="module" src="/components/product-detail/init.js"></script>` 한 줄로 끝내세요.

> 상세한 해결 방법은 `2026041509_리뷰_해결.md` **3-5** 참고

### 3-6. `add.js`와 `edit.js`에 똑같은 `createColor` 함수가 있어요 (중요도: 중간)

**파일 1**: `packages/admin/pages/components/product/add.js` 40~89줄
**파일 2**: `packages/admin/pages/components/product/edit.js` 9~57줄

거의 같은 50줄짜리 `createColor(value)` 함수가 두 파일에 복사되어 있어요. SVG 휴지통 아이콘 path까지 완벽하게 똑같아요.

### 3-8. `orderSummation.js`가 JS에서 인라인 스타일을 지정해요 (중요도: 중간)

```html
<!-- order.html의 container 요소 -->
<div class="order-summation-container max-h-0 overflow-hidden transition-[max-height] duration-300 ease-out">
  ...
</div>
```

```js
// orderSummation.js — 열 때
container.classList.remove("max-h-0");
container.classList.add("max-h-[500px]");
```

### 3-9. `arrivals.js`와 `bestSeller.js`의 이미지 선택 로직이 중복돼요 (중요도: 중간)

**파일 1**: `packages/user/components/slide/arrivals.js` 23줄, 49줄
**파일 2**: `packages/user/components/slide/bestSeller.js`

같은 파일 **안에서** 이미지 고르는 표현이 두 번 등장해요:
```js
src="${item.images?.[1] ?? item.images?.[0] ?? ""}"
```

이 "두 번째 이미지가 있으면 그걸 쓰고, 없으면 첫 번째를 쓴다"는 규칙이 **여러 파일, 여러 줄에 흩어져** 있어요.

**왜 개선하면 좋을까?**

나중에 "사실은 세 번째 이미지를 기본으로 쓰자" 같은 변경이 생기면, **몇 군데를 고쳐야 하는지 아무도 정확히 몰라요.** 이게 바로 응집도가 낮다는 얘기예요.

**추천:** `utils/productImage.js` 같은 파일에 `getDisplayImage(item)` 같은 작은 함수를 두세요.

```js
// utils/productImage.js
export function getDisplayImage(item) {
  return item.images?.[1] ?? item.images?.[0] ?? "";
}
```

```js
// arrivals.js
import { getDisplayImage } from "../../utils/productImage.js";
// ...
src="${getDisplayImage(item)}"
```

### 3-12. `imageApi.js`가 이미지를 하나씩 순차 업로드해요 (중요도: 낮음)

**파일**: `packages/admin/pages/API/product/imageApi.js` 4~11줄

```js
export async function uploadImages(files) {
  const urls = [];
  for (const file of files) {
    const formData = new FormData();
    formData.append("file", file);
    const result = await upload("/admin/images", formData);
    urls.push(result.data.imageUrl);
  }
  return urls;
}
```

**왜 개선하면 좋을까?**

파일이 5개면 5번 연속으로 기다려요. 업로드가 각 2초라면 총 10초가 걸려요. 관리자 입장에서 "어? 화면이 멈췄나?" 싶은 경험이 되죠.

**추천:** `Promise.all`로 병렬 업로드하면 가장 오래 걸리는 1~2초 만에 끝나요.

```js
export async function uploadImages(files) {
  const uploads = Array.from(files).map((file) => {
    const formData = new FormData();
    formData.append("file", file);
    return upload("/admin/images", formData);
  });
  const results = await Promise.all(uploads);
  return results.map((r) => r.data.imageUrl);
}
```

## 6. 마무리

**이번 모노레포 전환은 정말 큰 발전이에요.** `src/` 하나에 모든 걸 쑤셔 넣던 상태에서 `packages/user/`, `packages/admin/`, `packages/share-api/`로 명확하게 나눈 것, 그리고 공용 HTTP 모듈을 만들어낸 것. 이건 **실무 프로젝트의 기본 구조**와 거의 같아요. 교육 과정에서 이 정도까지 해낸 건 정말 대단합니다. 👏

이번 리뷰의 대부분은 **"크게 잘한 다음에 마무리가 살짝 덜 된 부분"** 을 가리키고 있어요. 뼈대는 훌륭해요. 살만 조금 다듬으면 됩니다.

한 가지만 기억해주세요:

> **"같은 코드가 두 곳에 있으면 하나로 합칠 수 있는지 먼저 고민해보자."**
> **"DOM을 만지기 전에 null인지 한 번 확인하자."**
> **"스타일은 JS가 아니라 Tailwind 클래스로."**

이 세 가지만 습관이 되면 이번 리뷰의 절반이 자동으로 해결돼요. 응원합니다! 🦁
