## 2-1. 모바일 메뉴 생성 책임을 한 곳으로 모으기

### 문제

현재는

- HTML이 `#mobileMenu`를 만들고
- JS도 `#mobileMenu`를 또 만들고 있습니다.

이 상태에서는 같은 `id`가 중복될 수 있고, 메뉴 애니메이션 코드가 어떤 요소를 잡는지 헷갈려집니다.

### 가장 쉬운 해결 방법

이미 여러 페이지에 빈 `<div id="mobileMenu"></div>`가 있으므로, **HTML은 컨테이너만 두고 JS는 그 안 내용만 채우기** 방식으로 맞추는 것을 추천합니다.

### Step 1. `createMobileMenu.js`에서 새 `div`를 만들지 말고 기존 컨테이너를 찾으세요

```js
// 수정 전
const mobileMenu = document.createElement("div");
mobileMenu.id = "mobileMenu";
mobileMenu.className =
  "fixed opacity-0 pointer-events-none transition-all duration-500 top-0 z-49 left-0 bg-white w-screen h-full";
```

```js
// 수정 후
const mobileMenu = document.getElementById("mobileMenu");
if (!mobileMenu) return;

mobileMenu.className =
  "fixed opacity-0 pointer-events-none transition-all duration-500 top-0 z-49 left-0 bg-white w-screen h-full";
mobileMenu.innerHTML = "";
```

### Step 2. `document.body.prepend(mobileMenu)`를 제거하세요

지금은 새 메뉴를 body 맨 앞에 끼워 넣고 있는데, 기존 컨테이너를 사용할 거라면 이 코드는 필요 없습니다.

```js
// 삭제할 코드
document.body.prepend(mobileMenu);
```

### Step 3. 기존처럼 내부 요소만 붙이세요

아래 코드는 그대로 써도 됩니다.

```js
aside.append(nav, accountDiv, wishlistDiv, utilDiv);
mobileMenu.appendChild(aside);
```

### Step 4. 페이지 HTML은 지금처럼 빈 컨테이너만 유지하세요

```html
<div
  id="mobileMenu"
  class="fixed opacity-0 pointer-events-none transition-all duration-500 top-0 z-49 left-0 bg-white w-screen h-full"
></div>
```

### 왜 이 방식이 좋은가요?

- 메뉴 상자 생성 책임은 HTML이 가집니다.
- 메뉴 내부 내용 생성 책임은 JS가 가집니다.
- 각 파일의 역할이 겹치지 않습니다.

즉, **결합도는 낮아지고 읽기도 쉬워집니다.**

## 2-2. `APIError` 대신 실제로 존재하는 값만 검사하기

### 문제

`wishlistApi.js`에서 `APIError`를 쓰고 있지만 import가 없습니다.

```js
if (err instanceof APIError && err.status === 401) return [];
```

이 코드는 에러가 났을 때 오히려 또 다른 에러를 만들 수 있습니다.

### 가장 쉬운 해결 방법

지금 `share-api`는 에러 객체에 `status`를 붙여서 던지고 있으므로, **그 값만 검사하면 충분합니다.**

### Step 1. `wishlistApi.js`를 아래처럼 고치세요

```js
import { get } from "@gentlelion/share-api";
import { getToken } from "../token/getToken";

export const getWishlist = () => {
  if (!getToken()) {
    return null;
  }

  return get("/wishlist").catch((err) => {
    if (err.status === 401) return null;
    throw err;
  });
};
```

### 왜 `null`을 추천하나요?

지금 코드 흐름에서는 “비로그인 상태라 위시리스트 데이터가 없음”을 표현하는 값이 필요합니다.

- `[]`는 “로그인했지만 데이터가 빈 배열”처럼 보일 수 있고
- `null`은 “아예 현재 데이터가 없음”을 표현하기 더 쉽습니다.

초보 단계에서는 이런 구분이 중요합니다.

> **빈 데이터와, 데이터 자체가 없는 상태는 다릅니다.**

## 2-3. 리다이렉트할 때는 함수도 바로 끝내기

### 문제

현재 `wishbuttonEvent.js`는 로그인 페이지로 보내면서도 함수 실행을 멈추지 않습니다.

```js
if (!data) {
  alert("로그인이 필요한 기능입니다.");
  window.location.href = "/components/login/login.html";
}
```

### 가장 쉬운 해결 방법

`return` 한 줄을 넣어서, 그 아래 로직이 실행되지 않게 만드세요.

### Step 1. 가장 먼저 `return`을 추가하세요

```js
if (!data) {
  alert("로그인이 필요한 기능입니다.");
  window.location.href = "/components/login/login.html";
  return;
}
```

이 한 줄만으로도 큰 문제가 줄어듭니다.

### Step 2. 조금 더 읽기 쉽게 나누고 싶다면 함수로 빼세요

```js
function requireLogin() {
  alert("로그인이 필요한 기능입니다.");
  window.location.href = "/components/login/login.html";
}
```

```js
button.addEventListener("click", async function () {
  if (!data) {
    requireLogin();
    return;
  }

  const productId = Number(button.dataset.productId);
  const isWishlisted = button.dataset.wishlist === "1";

  if (!isWishlisted) {
    const color = button.dataset.color || "";
    const result = await postWishlist(productId, color);
    if (!result) return;

    button.dataset.wishlist = "1";
    changeSVG(button, true);
    return;
  }

  const item = data.data.items.find(function (item) {
    return item.productId === productId;
  });

  if (!item) return;

  const result = await delWishlist(item.wishlistItemId);
  if (!result) return;

  button.dataset.wishlist = "0";
  changeSVG(button, false);
});
```

### 왜 이 방식이 좋은가요?

- 비로그인 분기가 바로 끝납니다.
- 로그인된 상태의 로직만 아래에 남아서 읽기 쉽습니다.
- 삭제할 항목이 없을 때도 안전하게 빠져나올 수 있습니다.

즉, **함수의 흐름이 명확해집니다.**

## 2-4. 테스트 산출물은 `.gitignore`에 넣고, 이미 추적 중인 파일은 인덱스에서 제거하기

### 문제

Playwright 리포트와 마지막 실행 상태 파일은 소스 코드가 아닙니다. 그런데 현재는 저장소에 포함되어 있습니다.

### Step 1. `.gitignore`에 아래 두 줄이 있는지 확인하세요

```gitignore
playwright-report/
test-results/
```

### Step 2. 이미 추적 중인 파일은 인덱스에서 제거하세요

```bash
git rm --cached -r playwright-report test-results
```

이 명령은 **로컬 파일 자체를 지우는 게 아니라**, Git 추적 대상에서만 빼는 작업입니다.

### Step 3. 이후 다시 테스트를 실행해도 결과물은 커밋하지 마세요

테스트를 돌린 뒤 PR에 들어가야 하는 것은 보통 아래 둘 중 하나입니다.

- 테스트 코드
- 테스트를 통과시키기 위한 실제 소스 변경

리포트 HTML 파일 전체가 PR에 들어가는 경우는 거의 없습니다.

### 왜 이 방식이 좋은가요?

- PR diff가 조용해집니다.
- 리뷰어가 진짜 코드 변경만 볼 수 있습니다.
- 팀 작업물이 더 깔끔해집니다.
